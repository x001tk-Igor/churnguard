## Какую боль закрывает {#какую-боль-закрывает}

### Со стороны Data Analyst (боль #4 и #6 из исследования)

> «Мы не знаем, кто из клиентов уйдёт, и теряем деньги на реактивных действиях.»

**Как проявляется:**
- Компания видит отток постфактум — когда клиент уже перестал платить
- Retention-менеджеры работают вслепую: звонят всем подряд, сжигая бюджет
- Нет сегментации: не различают VIP-клиента от случайного
- Данные есть (логины, покупки, тикеты), но никто не превращает их в прогноз

**Что ChurnGuard даёт:**
- Показывает **кого удерживать** → приоритизация по вероятности оттока + ценности клиента
- Показывает **почему уходят** → факторы риска для каждого сегмента
- Показывает **что делать** → конкретные retention-действия с прогнозом эффекта

### Со стороны AI/ML Engineer (боль #1 и #3 из исследования)

> «PoC-кладбище: модель в Jupyter есть, в проде — нет.»

**Как проявляется:**
- Data Scientist построил модель в ноутбуке — она не переживает реальные данные
- Нет MLOps: никто не мониторит дрифт, не переобучает, не версионирует
- Бизнес не доверяет модели: «почему ты думаешь, что этот клиент уйдёт?»

**Что ChurnGuard даёт:**
- Production-ready пайплайн: загрузка CSV → профилирование → обучение → predict → дашборд
- Explainability: SHAP-значения для каждого прогноза — видно, какой фактор тянет клиента к оттоку
- Автоматическое сравнение моделей (LogReg vs XGBoost vs RandomForest) с выбором лучшей
- Детектор дрифта данных: если распределение признаков поплыло — предупреждение

### Со стороны Business Analyst (боль #3 и #6 из исследования)

> «Войны данных: отдел маркетинга считает отток по-своему, продукт — по-своему.»

**Как проявляется:**
- Нет единого определения «оттока»: для SaaS это 30 дней без логина? Для e-com — 90 дней без заказа?
- Стейкхолдеры спорят о метриках вместо того, чтобы действовать
- Рекомендации не привязаны к бизнес-процессу: «удерживайте клиентов» — а как именно?

**Что ChurnGuard даёт:**
- Настраиваемый горизонт оттока (7/14/30/60/90 дней) — компания сама решает, что считать оттоком
- RFM-сегментация как BA-фундамент: Recency, Frequency, Monetary — язык, понятный бизнесу
- Бизнес-рекомендации на каждый сегмент: не «удерживайте», а «отправьте промокод на 15% клиентам из сегмента At Risk с Lifetime Value > $500»
- Одностраничный executive summary для CEO

## User Journey: что видит пользователь {#user-journey}

```

```

   - Lost (R↓↓ F↓ M↓) — 26%, 12% выручки → СЕРЫЙ
→ Интерактивная heatmap: наведи на ячейку — увидишь конкретных клиентов

### Почему Streamlit, а не Next.js?

| Критерий | Streamlit | Next.js + FastAPI |
|----------|-----------|-------------------|
| Скорость разработки MVP | **2–3 дня** | 2–3 недели |
| Интерактивные графики | Из коробки (plotly/altair) | Ручная интеграция |
| Data-фичи (кэш, загрузка CSV) | Из коробки | Кодить вручную |
| Демонстрация DA/ML-скиллов | **Прямой показ кода аналитики** | Код скрыт за API |
| Демонстрация инженерных скиллов | Слабо | Сильно |
| Деплой | 1 клик (Streamlit Cloud) | Настройка сервера |

**Решение:** ChurnGuard → Streamlit (показывает DA + ML скиллы). ReqForge → Next.js + FastAPI (показывает инженерные скиллы). Два проекта закрывают обе аудитории.

## ML-подход: модель оттока {#ml-подход}

### Пайплайн обучения

```
1. Загрузка и валидация схемы
2. Data profiling → Health Score → авто-исправления
3. Feature engineering:
   ├── lag-признаки (активность 7/14/30 дней назад)
   ├── ratio-признаки (тикеты / транзакции)
   ├── trend-признаки (растёт/падает частота логинов)
   └── категорные → OneHot / Ordinal
4. Train/test split (stratified, 80/20)
5. SMOTE для балансировки классов
6. Сравнение 3 моделей:
   ├── Logistic Regression (базовый бенчмарк, интерпретируемый)
   ├── Random Forest (ловит нелинейности)
   └── XGBoost (чемпион по табличным данным)
7. Auto-select: выбор модели с лучшим AUC-ROC
8. Калибровка вероятностей (Platt scaling)
9. SHAP-анализ: топ-факторы + waterfall для каждого клиента
10. Порог вероятности: настройка по business-метрике (cost of false positive vs false negative)
```

### Метрики, которые показывает система

| Метрика | Что измеряет | Почему важно для бизнеса |
|---------|-------------|-------------------------|
| AUC-ROC | Качество ранжирования | Насколько хорошо модель отделяет уходящих от остающихся |
| Precision | Сколько из помеченных «уйдёт» реально ушли | Не тратим деньги на ложные тревоги |
| Recall | Сколько реально ушедших мы нашли | Не пропускаем уходящих клиентов |
| F1 | Баланс Precision/Recall | Единая метрика качества |
| Lift @ 10% | Во сколько раз модель лучше случайной на топ-10% | Retention-менеджер работает с верхушкой списка |
| Money Saved | $ сумма LTV клиентов, которых удержали бы | Главная бизнес-метрика |

### Почему не deep learning?

Табличные данные + 5K–50K строк → XGBoost даёт лучшее качество, чем нейросети, и тренируется за секунды, а не часы. Плюс SHAP для деревьев — зрелая библиотека. DL здесь был бы оверинжинирингом.

## BA-слой: бизнес-логика и рекомендации {#ba-слой}

### RFM-сегментация (методология)

```
RFM-скор присваивается каждому клиенту:
- Recency (R): квартиль по days_since_last_activity (чем меньше, тем лучше)
- Frequency (F): квартиль по transaction_count (чем больше, тем лучше)
- Monetary (M): квартиль по total_revenue (чем больше, тем лучше)

Каждый параметр: 1 (худший) → 4 (лучший)
Итого 4×4×4 = 64 RFM-ячейки → агрегируются в 5 бизнес-сегментов
```

### 5 сегментов и действия

| Сегмент | RFM | % клиентов | Действие | Канал | Тон |
|---------|-----|-----------|----------|-------|-----|
| **Champions** | R4 F4 M3-4 | ~12% | Не трогать. Программа лояльности. Попросить рефералов. | Email | Благодарность |
| **Loyal** | R3-4 F2-3 M2-3 | ~23% | Upsell. Новые фичи. Кейс-стади. | In-app | Вовлечение |
| **At Risk** | R1-2 F3-4 M3-4 | ~8% | 🔥 **Критично.** Звонок менеджера. Промокод 20%. Спецпредложение. | Телефон + Email | Срочность |
| **Hibernating** | R1-2 F1-2 M1-3 | ~31% | Реактивация. «Мы скучаем». Опрос NPS. Показ новых фичей. | Email-цепочка | Теплота |
| **Lost** | R1 F1 M1-2 | ~26% | Win-back кампания. Глубокий дисконт или не тратить бюджет. | Email | Последний шанс |
